\chapter*{Résumé}
\addcontentsline{toc}{chapter}{Résumé}
Ce rapport présente \textbf{Mon PF App}, une application mobile de gestion des commandes et des livraisons d'un restaurant. Le projet répond à la problématique de la gestion manuelle : erreurs de commande, retards de préparation, manque de suivi et communication insuffisante entre client, restaurant et livreur.

La solution s'appuie sur \textbf{Flutter} pour l'application mobile et \textbf{Laravel} pour l'API backend. Le système permet au client de consulter le menu, gérer le panier, passer une commande et suivre son état. Il propose aussi des espaces dédiés à l'administrateur, au serveur et au livreur.

La version livrée est une version \textbf{démo-prod} : elle dispose d'une interface aboutie, d'un backend local fonctionnel, d'une base SQLite seedée, de tests automatiques et d'une documentation permettant d'installer, tester, debugger et présenter le projet.

\chapter{Contexte et problématique}
Les restaurants gèrent de plus en plus de commandes à emporter et en livraison. Lorsque les commandes sont prises manuellement, les erreurs de saisie, les oublis et les retards deviennent fréquents.

Le client souhaite connaître l'état de sa commande. Le restaurant doit organiser les préparations. Le livreur doit savoir quelles commandes sont disponibles et lesquelles lui sont affectées.

La problématique du projet est donc la suivante :
\begin{quote}
Comment digitaliser le processus de commande afin de réduire les erreurs humaines, accélérer le traitement et améliorer la communication entre les acteurs ?
\end{quote}

\begin{longtable}{p{4cm}p{5cm}p{5cm}}
\caption{Problèmes identifiés et réponses du projet}\\
\toprule
\textbf{Problème} & \textbf{Conséquence} & \textbf{Réponse Mon PF App}\\
\midrule
Erreur de commande & Plat incorrect ou quantité fausse & Panier numérique et validation explicite\\
Retard de préparation & Client insatisfait & Suivi par statut\\
Mauvaise communication & Livreur ou restaurant mal informé & Espaces dédiés par acteur\\
Menu difficile à maintenir & Prix ou disponibilité non à jour & Catalogue API et endpoints de gestion menu\\
\bottomrule
\end{longtable}

\chapter{Objectifs du projet}
L'objectif général est de réaliser une application mobile de gestion des commandes d'un restaurant. Elle doit organiser le processus depuis le choix des plats jusqu'à la livraison.

Les objectifs spécifiques sont :
\begin{itemize}
  \item permettre aux clients de consulter le menu ;
  \item ajouter des plats au panier ;
  \item calculer automatiquement le montant total ;
  \item passer une commande ;
  \item suivre l'état de la commande ;
  \item permettre au livreur de consulter et accepter les livraisons ;
  \item permettre à l'administrateur de consulter les statistiques et les commandes ;
  \item structurer le projet de manière professionnelle pour faciliter la maintenance.
\end{itemize}

Sur le plan technique, le projet doit être testable et démontrable. Le repository a donc été organisé avec un frontend Flutter, un backend Laravel, des documents de spécification, des guides et des scripts.

\chapter{Sécurité}
La sécurité a été renforcée à plusieurs niveaux.

\begin{itemize}
  \item la sélection de rôle a été retirée de l'inscription publique ;
  \item le backend force toujours le rôle \texttt{client} lors d'une inscription ;
  \item le token est stocké dans \texttt{flutter\_secure\_storage} côté application ;
  \item l'API ne stocke que le hash SHA-256 du token ;
  \item les appels API possèdent un timeout ;
  \item Android release refuse HTTP local et doit utiliser HTTPS.
\end{itemize}

\chapter{Algorithmes et règles métier}
Le projet repose sur des algorithmes applicatifs simples et défendables.

\section{Gestion du panier}
Lorsqu'un utilisateur ajoute un plat au panier, l'application recherche si ce plat existe déjà dans la liste. S'il existe, la quantité augmente. Sinon, une nouvelle ligne \texttt{CartItem} est créée.

\section{Calcul du total}
Chaque ligne calcule son total par :
\[
  total\_ligne = prix\_unitaire \times quantite
\]
Le total de la commande est ensuite calculé par :
\[
  total = sous\_total + frais\_livraison
\]

\section{Règles de commande}
\begin{itemize}
  \item une commande ne peut pas être créée sans plats ;
  \item les quantités doivent rester positives ;
  \item le statut est normalisé par le backend ;
  \item les rôles déterminent les actions autorisées.
\end{itemize}

\chapter{Android et Windows}
Android est la cible la plus avancée. L'APK debug existe et peut être installé avec ADB lorsque le téléphone est connecté et autorisé.

\begin{lstlisting}[style=monpf]
C:\Users\LOQ\AppData\Local\Android\Sdk\platform-tools\adb.exe devices
C:\Users\LOQ\AppData\Local\Android\Sdk\platform-tools\adb.exe install -r build\app\outputs\flutter-apk\app-debug.apk
\end{lstlisting}

Windows est une cible prioritaire, mais le build nécessite l'installation du composant Visual Studio ATL. L'erreur observée est :

\begin{lstlisting}[style=monpf]
fatal error C1083: Impossible d'ouvrir le fichier include : 'atlstr.h'
\end{lstlisting}

La solution est d'ouvrir Visual Studio Installer et d'installer le composant \textbf{C++ ATL for latest v143/v14x build tools} ou l'équivalent disponible.
